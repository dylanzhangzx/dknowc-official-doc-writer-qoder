# 任务路由与模式选择

本文件用于帮助 Agent 判断何时直接写作、何时追问、何时搜索、何时进入严格流程。

## 先识别任务类型

1. 起草：用户要求写、起草、撰写、生成某类公文或材料。
2. 润色改写：用户提供已有文本，要求优化、压缩、扩写、改成正式公文。
3. 审查把关：用户要求检查格式、逻辑、政策依据、语言或风险。
4. 搜索支撑：用户要求找政策、依据、数据、案例。
5. Word 交付：用户要求生成 `.docx`、红头文件或正式排版文件。

## 再判断复杂度

简单任务：

- 会议通知、时间地点变更、内部事务通知
- 简短告知函、提醒函
- 用户信息完整、无需政策依据

处理方式：读取必要文种标准后直接生成。短文本可在聊天中返回；如用户要求文件或正式交付，则生成 Word。不要制造多轮确认。

常规任务：

- 通知、函、短报告、普通会议纪要
- 用户给出主题、对象、事项和基本要求

处理方式：必要时追问 1-3 个关键缺失信息。短文本可直接生成正文；报告、总结、方案、汇报材料等长篇正式材料默认生成 Word，不先在聊天中发送正文初稿。

复杂任务：

- 请示、复函、政策依据型报告
- 调研报告、实施方案、管理办法、工作总结、汇报材料
- 产业研究总结、产业发展报告、政策研究报告、年度发展总结
- 工作要点、专项整治方案、会议讲话、研讨发言等长篇材料
- 涉及政策、数据、案例或多章节结构

处理方式：按需进入搜索、素材确认、大纲确认和审查；最终默认交付 Word，除非用户明确要求只看正文。执行过搜索时，正式 Word 与素材来源说明分开交付。

高风险任务：

- 引用政策依据
- 回复来函、作出承诺或表达正式意见
- 涉及跨地区政策适用、数据真实性、红头文件

处理方式：必须执行素材来源检查和生成前审查。

## 五种模式如何组合

Tool Wrapper：

- 深知搜索脚本
- Word 排版脚本
- 红头生成脚本
- 素材来源说明文件生成

Generator：

- 默认写作模式
- 使用文种标准、用户材料和 `reference/material_usage_guidance.md` 生成内部正文草稿；长篇正式材料不得把草稿直接发给用户
- 对工作总结、工作要点、实施方案、专项整治方案、会议讲话、研讨发言、汇报材料等长篇材料，生成前先内部完成文体判断、结构选择、小标题策略和段落功能分配
- 可参考 `reference/standards/99_可选参考_常用句式和结构.md`，但不得把参考句式机械套用成固定模板；长篇材料应先保证任务闭环，再考虑表达增强
- 对执行过搜索的任务，正式正文不得包含素材清单、知识专库链接或 URL；另生成素材来源说明

Reviewer：

- 生成前检查
- 生成后自检
- 用户要求“帮我看看有没有问题”时优先使用

Inversion：

- 需求不完整且无法合理推断时
- 长篇复杂材料开始前
- 一次最多问 3 个关键问题

Pipeline：

- 执行过搜索的复杂公文
- 政策依据型材料
- 红头文件生成
- 需要用户确认素材或大纲的任务

## 决策规则

- 能直接完成的简单任务，直接完成。
- 信息缺失但可合理假设，先写并标注假设，不要频繁打断。
- 信息缺失会影响公文方向、对象、结论或风险时，先追问。
- 需要政策、数据、案例支撑时，遵循 `reference/search_policy.md`，先展示搜索方案并等待用户确认。
- 短公文没有明确依据需求时，默认不做大规模检索；调研报告、政策研究报告和产业研究报告才使用较完整的政策、数据、案例检索。
- 用户确认搜索方案后，只能调用深知搜索脚本 `scripts/dkag_search.py`；不得使用 Web Search、Web Fetch、浏览器搜索或公开网页抓取替代。
- 深知搜索异常、空结果或素材不足时，先暂停请用户确认下一步，不得自行切换外部搜索。
- 用户明确要求 Word，或任务属于报告、总结、方案、汇报材料、产业研究等长篇正式材料时，正文完成并检查后只调用普通 Word 排版脚本；如生成临时 Markdown 草稿，必须直接继续生成 `.docx`，不得先把正文或 `.md` 发给用户。
- 只有用户明确要求红头文件、红头版、套红头或生成红头时，才在普通 Word 基础上调用红头脚本。

## Word 任务交付路径

涉及长篇报告、总结、方案、汇报材料、产业研究、政策研究，或用户明确要求 Word 时，路径固定为：

1. 确认必要信息或搜索方案。
2. 执行深知搜索并整理素材。
3. 判断文体结构和段落功能，必要时参考 `reference/standards/99_可选参考_常用句式和结构.md`。
4. 按 `reference/material_usage_guidance.md` 判断材料用途和风险，不做机械拼贴。
5. 生成内部正文草稿。
6. 必要时保存临时 Markdown 供脚本读取；正文中不得附带素材使用情况和知识专库链接。
7. 立即生成最终 `.docx`。
8. 如执行过搜索，单独生成 `标题_素材来源说明.docx`；只有用户明确要求纯文本或 Markdown 时，才可改为 `.md`。
9. 向用户交付正式 `.docx` 文件路径；如有素材来源说明，一并说明其为辅助溯源文件，不是正文附件。

不得在第 3-5 步向用户发送正文初稿、正文全文、Markdown 文件链接或中间文件路径。需要用户审阅时，应先交付 Word，再请用户基于 Word 提修改意见。

如果用户只说“写一份”“起草”“生成”“整理成”某类长篇正式材料，默认也走上述 Word 路径；不要把“未明确要求 Word”理解为需要聊天正文交付。
