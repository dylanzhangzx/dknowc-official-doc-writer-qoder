# 公文生成审查清单

用于 Reviewer 模式。检查应简洁、按风险优先，不要把简单任务变成冗长审稿。

## 通用检查

- 文种是否匹配用户目的。
- 标题是否准确、简洁、正式。
- 主送机关、正文、落款、日期是否完整。
- 结构层级是否清晰，是否符合对应文种标准。
- 语言是否正式、稳妥、无口语化表达。
- 是否存在事实、数据、政策名称或文号不确定却写成确定结论。
- 政策名称、文号、发布日期、精确数字、排名、占比、金额和“全国首个/唯一/领先”等高风险表述是否均有明确来源支撑。
- 超出用户题目时间范围的信息是否仅作为背景、延续动态或趋势参考，未混作当期政策或当期成效。
- 是否出现不必要的承诺、绝对化表述或越权表述。
- 对工作总结、工作要点、实施方案、会议讲话、研讨发言、汇报材料等长篇材料，是否先形成清晰结构，而不是把素材摘要顺序拼接。
- 小标题是否能准确概括对应段落，是否避免标题很正式但正文没有具体内容。
- 正文是否形成任务闭环：目标、任务、措施、责任/时限、保障至少覆盖其应有部分。
- 是否存在只有排比句、口号和抽象表态，没有具体任务、对象或抓手的段落。
- 问题短板是否稳妥，未把未经用户确认的问题写成用户单位确定存在的严重问题。

## 搜索素材检查

- 政策依据型素材是否只使用国家级和本省来源。
- 数据支撑型素材是否只使用国家级和本省来源。
- 其他省市材料是否仅作为参考案例，不作为本省政策依据。
- 表述参考是否只从已召回材料中吸收表达风格，且没有作为独立搜索项或独立素材类型列入。
- 执行过搜索时，正式公文正文是否未包含 `【素材使用情况】`、`【知识专库链接】`、知识专库 URL 或素材清单。
- 是否已单独生成 `标题_素材来源说明.html`，且没有生成 Word 版素材来源说明。
- 素材来源说明 HTML 开头是否说明知识专库链接可用于回看召回材料、核验来源出处和追溯正文依据。
- 素材来源说明 HTML 中每条核心素材是否以卡片形式标注来源、正文对应、支撑内容，必要时补充核验提示，而不是只罗列材料标题。
- 素材来源说明 HTML 是否未使用表格，未出现“表1 主要情况汇总”等兜底表题，而是使用卡片式溯源布局。
- 素材来源说明是否没有只用 `用途：` 代替 `来源：`、`正文对应：`、`支撑内容：`。
- 素材来源说明中每条素材是否使用正式格式，且没有 `→ 用于：`、箭头符号等内部工作流标记。
- 素材来源说明是否只列实际用于正文的核心材料，同类材料是否已合并说明，未机械罗列所有搜索结果。
- 素材来源说明中素材来源是否无重复前缀，如 `来源：来源：`、`数据源：数据源：`。
- 素材来源说明 HTML 是否包含清晰的知识专库链接按钮，按钮文字来自搜索目的，底层超链接保留原始 URL。
- 知识专库链接是否逐条来自原始搜索结果 JSON 的 `knowledgeBase` 字段，未被手写、猜测、改写或混用。
- 知识专库链接的外显文字是否优先使用对应搜索结果 JSON 的 `search_meta.purpose`，即用户确认搜索方案中的“搜索目的”，而不是泛化显示为“知识专库链接1/2/3”。
- 知识专库链接标签是否为纯文本，未带编号、Markdown 加粗符号或预写的“（点击打开）”。
- 政策依据型检索是否使用 `--policy`，或是否说明未使用原因。
- 搜索异常、空结果或关键素材缺失时，是否已暂停并征得用户确认下一步。
- 是否全程使用深知搜索脚本完成素材检索，而不是 Web Search、Web Fetch、浏览器搜索或公开网页抓取。
- 是否避免未经用户同意自动改用 Web Search、Web Fetch 或公开官网检索。
- 用户确认搜索方案时，是否未展示 `--area`、`--search-type`、`--policy`、`--search-channel`、`--clean`、`--output` 等内部脚本参数，且未设置“参数”列。
- 是否按 `reference/material_usage_guidance.md` 将政策、数据、案例和表述参考分别转化为正文用途，未把搜索摘要简单拼贴成正文。
- 是否避免用户口径被搜索材料覆盖，尤其是单位、地域、时间、对象和工作要求。
- 是否避免将外地案例、本地政策、国家政策、旧年份数据混写为同一层级结论。
- 短公文是否避免不必要的大规模检索和政策堆砌。
- 调研报告、政策研究报告和产业研究材料是否形成“事实支撑-问题判断-原因分析-对策建议”的链条。

## 文种专项检查

请示：

- 是否一文一事。
- 是否有明确请示事项。
- 是否避免多头主送。
- 是否避免在请示中夹带报告事项。

报告：

- 是否以陈述情况、反映问题、提出建议为主。
- 是否避免夹带请示事项。
- 总结报告类是否区分成绩、问题和下一步安排，避免三者混写。
- 下一步安排是否回应前文问题和用户目标，避免泛泛而谈。

工作要点：

- 是否体现年度或阶段性任务安排，未写成已完成工作的总结。
- 是否包含总体要求、重点任务和保障措施。
- 重点任务是否有明确动作、对象、抓手和预期效果。

实施方案/专项整治方案：

- 是否说明背景依据、工作目标、重点任务、阶段安排、责任分工和工作要求。
- 是否体现问题导向、清单管理、分类整改、闭环销号等执行机制。
- 是否避免只有原则性要求，缺少阶段、责任和验收安排。

讲话/研讨发言：

- 是否有清晰开场，说明会议背景、主要任务或发言主题。
- 是否符合“形势/成绩-问题-部署-责任”或“认识-重点-保障”的逻辑。
- 部署要求是否具体，避免整篇只有动员表态。

函/复函：

- 是否逐项回应来函事项。
- 是否避免使用“逾期视为无意见”等不当表述，除非用户明确要求且适用。
- 是否避免越权承诺。

通知：

- 时间、地点、对象、事项、要求是否清楚。
- 执行要求是否具体、可操作。

会议纪要：

- 会议时间、地点、主持、参会、议题、议定事项是否清楚。
- 是否区分汇报内容和议定事项。

## Word 输出检查

- 是否只使用允许的 Markdown 结构生成 Word。
- 是否避免列表、粗体、斜体、代码块、引用、分割线等不支持格式。
- 普通 Word 是否只包含正式正文、必要附件、落款、版记和固定 AI 生成提示，未混入素材溯源内容。
- 初始化检查如显示缺少公文标准字体，Word 交付说明是否提醒用户在安装标准字体的 Word/WPS 环境复核字体、分页和表格换行。
- 红头文件是否已移除普通 Word 中的 AI 生成提示，避免挤占或破坏国标版记中印发单位、印发日期的当页排版。
- 素材来源说明是否通过 `scripts/source_note_html.py` 生成 HTML，而不是用 Word 排版脚本或手写页面临时拼接。
- 用户要求红头时，是否先生成普通 Word，再使用代码化红头脚本生成红头文件。
- 用户未明确要求红头时，是否避免生成红头文件。
- 是否避免主动提及或生成 PDF；用户明确要求 PDF 时，是否只交付最终 Word 或红头 Word，并提示用户使用本机 Word/WPS 自行另存或导出为 PDF。
- 是否避免使用 LibreOffice、ReportLab、PyPDF2/pypdf、浏览器打印、HTML 转 PDF 或其他降级方案生成正式公文 PDF。
- 执行深知搜索前，是否已向用户展示搜索方案并获得确认。
- Word 交付时是否优先返回正式 `.docx`，未同时发送 Markdown 草稿、中间文件或长篇正文；如返回素材来源说明 HTML，是否说明其为辅助溯源文件。
- 生成 Word 时，是否先将多行正文写入临时正文文件并把文件路径传给排版脚本，未把整篇正文直接塞进 `--text` 参数，也未绕过脚本手写 `python-docx`。
- 如生成了临时 Markdown，是否未向用户展示、链接或发送，而是直接继续生成最终 `.docx`。
- 对报告、总结、方案、汇报材料、产业研究等长篇正式材料，是否默认生成 Word，而不是先在聊天中发送正文初稿。
- 普通正文中的连续数字编号（如 `1.`、`2.`、`3.`）是否缩进和对齐一致，未把第 2 项以后误排成带 8 个手动空格的附件清单续行。
- 如包含表格，表格是否确实用于重复记录、指标对比、政策维度比较、责任分工、时间安排、问题清单等行列数据，而不是把普通论述硬塞进表格。
- 表格是否未直接作为小节标题使用，而是归入统一章节编号体系中的汇总性、比较性或综述性小节内。
- 每张表格是否都有表题（如“表1 XXX”），且表题作为普通文字出现在小节标题下方，未使用 `#`、`##`、`###` 或 `####` 标题语法。
- 多个表格编号是否全文连续（表1、表2、表3……），未按章节重新编号。
- 宽表是否已触发横向 A4 页面，或已拆分为多个竖版表格、转为附件，避免列宽过窄和文字拥挤。
- 跨页表格是否未重复表头，且同一行、同一单元格内容是否尽量未被拆到两页；如单元格内容过长，是否已改为正文段落、分条说明或附件。

## 输出审查结果格式

如发现问题，按严重程度输出：

```text
【审查结果】

高风险：
1. ...

中风险：
1. ...

低风险：
1. ...

建议修改：
1. ...
```

如未发现明显问题：

```text
未发现明显格式或内容风险。仍需人工确认政策依据、数据真实性和最终发文权限。
```
