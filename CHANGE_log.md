## [3.0.1] - 2026-05-19

### OpenClaw 实测反馈修复

#### 一、红头文件生成触发条件收紧

**问题：** 用户只要求生成 Word 时，Agent 可能同时生成普通 Word 和红头文件。  
**修复：** 明确默认只生成普通 Word；只有用户明确提出“红头文件”“红头版”“套红头”或“生成红头”时，才调用红头脚本。

#### 二、深知搜索前置确认恢复

**问题：** 涉及深知搜索的任务可能跳过搜索方案确认，直接检索并生成结果。  
**修复：** 强化 `SKILL.md`、`reference/search_policy.md` 和 `reference/task_router.md`：执行搜索前必须展示搜索地域、搜索内容、素材类型和使用边界，并等待用户确认或调整。

#### 三、知识专库链接清洗修复

**问题：** 服务器环境可能将搜索接口地址 `open.dknowc.cn/dependable/search/` 或括号说明写入【知识专库链接】。  
**修复：**
- `dkag_search.py` 不再把接口地址写入 `search_meta`
- 搜索异常信息不再暴露接口 URL
- `format_document.py` 过滤搜索接口地址，避免误写入 Word
- URL 后缀括号说明会在生成超链接前清理

#### 四、深知搜索时间参数容错

**问题：** `--policy` 搭配 `--time 2023-2025` 等横杠式年份范围时，深知搜索接口可能返回 `ret=500 / Internal Server Error`；清洗模式下还可能被误判为“未检索到相关参考文章”。  
**修复：**
- `dkag_search.py` 遇到 `2023-2025`、`2023年-2025年` 等范围时间时自动忽略时间过滤，避免触发接口 500
- 接口返回 `ret/errcode` 异常时直接输出“深知搜索接口返回异常”，不再清洗成空结果
- `SKILL.md` 和 `reference/search_policy.md` 明确：`--time` 只用于单个明确时间点；多年范围应省略时间或拆成单年检索

#### 五、按接口文档补充写作场景参数

**问题：** 深知搜索接口文档说明 `simplified` 默认值为 `true`，会按匹配度和发布日期剔除材料；但公文写作更适合先保留更多候选素材，再由 Agent 做四分类筛选。  
**修复：**
- `dkag_search.py` 固定传入 `simplified=false`
- 保留 `knowBase=true`，用于继续获取知识专库链接
- `SKILL.md` 和 `reference/search_policy.md` 补充固定参数说明

#### 六、异常处理改为用户确认

**问题：** 搜索或生成步骤异常时，Agent 可能自行跳过搜索、改用 Web 搜索或继续生成，导致来源边界和交付结果不可控。  
**修复：**
- `SKILL.md` 增加关键步骤异常暂停规则
- `reference/search_policy.md` 明确搜索异常、空结果、关键素材缺失时必须向用户确认下一步
- `reference/output_guide.md` 明确 Word 生成失败或格式异常时必须暂停确认
- `reference/review_checklist.md` 增加异常处理和未经同意改用 Web 搜索检查项

#### 七、搜索分类和 Word 交付规则收紧

**问题：** 测试中出现表述参考被设计成独立搜索项、长文正文在对话中截断、Word 交付同时附带 Markdown、知识专库链接与搜索 JSON 不一致等问题。  
**修复：**
- 表述参考不再作为独立搜索类型，只能从已召回材料中吸收表达方式
- 长篇公文默认直接生成 Word，不在对话中完整粘贴正文
- 交付时只返回最终 `.docx` 文件，不同时发送 Markdown 草稿或中间文件
- 政策依据型检索默认使用 `--policy`
- 知识专库链接必须来自原始搜索结果 JSON 的 `knowledgeBase` 字段
- `merge_search_results.py` 在合并结果中保留每次搜索的 `knowledgeBase`
- `format_document.py` 自动把 `素材使用情况`、`知识专库链接` 规范为 `【素材使用情况】`、`【知识专库链接】`

#### 八、强制深知搜索工具边界

**问题：** OpenClaw 测试中 Agent 绕过深知搜索脚本，直接调用 Web Search/Web Fetch 检索公开网页，导致素材来源、知识专库链接和交付行为失控。  
**修复：**
- `SKILL.md` 明确所有政策、数据、案例、素材检索默认只能使用 `scripts/dkag_search.py`
- `reference/search_policy.md` 新增“搜索工具边界”，禁止默认使用 Web Search、Web Fetch、浏览器搜索或公开网页抓取替代深知搜索
- `reference/task_router.md` 强化用户确认搜索方案后只能执行深知搜索脚本
- `reference/review_checklist.md` 增加是否绕过深知搜索的审查项

**结果：**
- 普通 Word 与红头 Word 的触发边界更清晰
- 涉及搜索的写作任务恢复“先确认搜索方案，再执行搜索”的交互节奏
- 知识专库链接区只保留真实知识专库 URL
- 政策检索不再因横杠式时间范围导致任务误判失败
- 搜索召回更贴合写作场景，减少可用材料被接口提前剔除的问题
- 搜索和文件生成异常时，Agent 会先请用户选择重试、调整、跳过或改用其他来源，不再自行决策
- Word 交付更聚焦，素材附录和知识库链接更稳定可追溯
- 检索链路默认固定为深知搜索，外部搜索必须经用户明确授权

#### 九、禁止中间 Markdown 交付

**问题：** WebChat 测试中，Agent 虽然最终生成了 Word，但先把内部 Markdown 草稿作为阶段性成果发给用户，导致交付链路不符合“只交付 Word”的要求。  
**修复：**
- `SKILL.md` 明确 Markdown 草稿只能作为内部临时文件，不得展示、链接、发送或要求用户审阅
- `reference/output_guide.md` 明确生成临时 Markdown 后必须继续生成 `.docx`
- `reference/task_router.md` 增加 Word 任务固定交付路径：内部草稿 → 临时 Markdown → 最终 Word → 只交付 `.docx`
- `reference/review_checklist.md` 增加是否误发 Markdown 草稿的检查项

**结果：**
- 长篇公文不会再先把 Markdown 草稿发给用户
- 用户明确要求 Word 时，默认只看到最终 `.docx` 文件

#### 十、长篇正式材料默认 Word 交付

**问题：** 用户只说“写一份/整理成/偏产业研究总结”等表达时，Agent 可能认为用户只是要聊天正文，于是先输出“正文初稿”。  
**修复：**

## [2.5.0] - 2026-03-25

### 素材交付与参考范文机制重构

**背景：** 四模型写作测试发现，素材一股脑丢给写作模型、不标注用途，导致写作效果不佳。同时发现深知搜索 `--full` 全文召回与 `--clean` 清洗函数存在冲突。

#### 一、参考范文召回（3.7节重写）

**旧流程：** "全文召回"——获取文章全文，没有区分"写作素材"和"成文参考"的用途

**新流程："参考范文召回"**
- 明确定位：普通搜索收集"写作素材"，参考范文召回获取"成文参考"（结构和逻辑）
- 用文章标题做query，**不加 `--clean`**（加了会丢弃全文字段）
- 从API返回的多篇结果中按标题精确匹配，只提取目标文章的"全文"字段
- 向用户推荐时说明参考价值（同类公文、结构完整、深度分析）
- 限制最多2篇，尽量控制在1篇
- 提取失败时告知用户，允许重新选择或跳过

#### 二、新增素材交付准备（3.8节）

**在用户确认素材、参考范文召回完成后、进入撰写前，AI将所有素材整理成结构化的"素材使用指引"**

- 写作素材按类型分组（政策依据、数据支撑、参考案例、表述参考），每组标注用途（→用于：正文某部分）
- 参考范文单独列出，标注参考价值、注意事项、全文内容
- 写作素材和参考范文严格分开，告诉写作模型各用来做什么
- 不要一股脑丢原始素材，经过分类和用途标注后再传给写作模型

#### 三、引用规范重写（5.0节）

**旧流程：** 正文加 `[^N^]` 引用标记，末尾列【参考来源】

**新流程：**
- 正文中**不加任何引用标记**，保持正文干净
- 所有素材使用情况统一在末尾【素材使用情况】中标注
- 【素材使用情况】分为两个子节：
  - 写作素材：每条标注"→ 用于：正文某部分"
  - 参考范文：标注"→ 参考：学习了哪些方面"（如有全文召回）
- 参考范文只学结构不照搬，旧数据需用写作素材中的新数据替换

#### 四、配套调整

- 6.3生成前检查：去掉引用标记检查，改为素材使用情况检查
- 3.4素材分类：去掉引用编号相关描述，素材分类表中"是否引用"改为"末尾标注"
- 3.4素材分类标题：去掉"需引用"字样

#### 五、`--full` 参数验证结论

| 发现 | 结论 |
|------|------|
| `--full` + `--clean` | 全文字段被清洗函数丢弃 |
| `--full` 不加 `--clean` | 全文字段正常返回 |
| API返回多篇全文 | 需按标题精确匹配，只提取目标文章 |
| 搜索引擎排序 | 时间优先，旧文章（如2024年）排得很后 |
| 全文召回token消耗 | 19篇全文约6.7万字符，无关全文会干扰写作质量和模型注意力 |

## [2.3.2] - 2026-03-17

### 日期匹配优化

**问题：** 落款日期使用占位符格式（如 `2026年【X】月【X】日`）时，正则无法匹配，导致日期排版不生效（右对齐）

**根因：** 原正则 `^\d{4}年\d{1,2}月\d{1,2}日$` 只匹配纯数字，`\d{1,2}` 无法匹配 `【X】`

**解决方案：**
1. 扩展正则，兼容【】占位符格式：`^\d{4}年(?:\d{1,2}|【[^】]+】)月(?:\d{1,2}|【[^】]+】)日$`
2. 添加位置约束：仅在文档后60%的行中才进行日期匹配，避免正文中的独立日期（如 `2026年1月1日`）被误判为落款日期

**修改文件：** `scripts/format_document.py` 第472行

def add_fullwidth_space_if_needed(text):
